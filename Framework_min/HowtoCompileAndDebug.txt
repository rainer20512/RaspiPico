How to program/debug RP2040 target with Rpi Debug Probe          RHB 17.11.2025
-------------------------------------------------------------------------------

0) Dieses HowTo gilt für Crossworks Version 5.X!!!
   in Version 4.X ist der RP 2040 und die PicoProbe noch sehr stiefmütterlich
   behandelt und mit vielen Anpassungen verbunden, s. unten

1) Im Crossworks-Paketmanager das Paket für PR2040 installieren und ein neues Projekt
   für RP2040 erstellen. "A C/C++ Executable for RaspPi RP2040" oder
   "A C/C++ Executable for RaspPi Pico". Alle anderen Einstellungen auf default lassen.
   Die notwendigen Systemfiles und ein leeres main.cpp werden hinzugefügt und das Projekt
   sollte sich jetzt shon mal fehlerfrei übersetzen und runterladen lassen.

2a.Flashen/Debuggen mit der Pico-Debug-Probe. Die erscheint als CMSIS-DAP unter den
   Targets. SWDIO, GND und SWCLK vom Target-Pico müssen mit der Pico-Probe verbunden 
   werden:  or=SWCLK, ye=SWDIO, sw=GND

2b.Wenn man den RP2040 so richtig zerschossen hat, ist keine Kommunikation/Programmierung
   über die PicoProbe mehr möglich. das zeigt sich daran, das beim flashen die Flash-Zellen
   des Target alle 0x00 sind und der Verifier meckert.
   Zur Abhilfe ein frischen UF2-File per Boot-Mode/USB auf den PR2040 kopieren. Ein 
   passendes UF2 liegt hier im Verzeichnis __Rescue__.

3a.Aus dem RasPi git-repo das Beispiel-Verzeichns fürs SDK "examples" runterladen.
   
3b.Das SDK für den PR240 ist im Paket enthalten, muss also nicht gesondert insalliert werden. 
   Allerdings sind nicht alle include-Pfade enthalten, da das Pico_SK sehr merkwürdig aufgebaut 
   ist, ie. fast immer nur ein header file in einem speziellen Verzeichnis. 

4. Jetzt das generierte main.cpp im Projekt durch das einfachste Beispiel aus den Beispielen
   "blink.c" ersetzen. 

5. in den Projekt-Eigenschaften das voreingestellte flash-Placement xml-file aus dem 
   Crossworks-Verzeichnis durch eine zum SDK passende, eigene Kopie ersetzen, 
   die hier abgelegt ist: flash_placement2040.xml

6. Jetzt wird es aufwändig: Da im Pico-SDK fast alle includes in einem eigenen Subdir
   liegen, muss in den Projekt-Einstellungen/Preprocessor der User-Include-Pfad
   iterativ mit dem entsprechenden Verzeichnis aus dem Pico-SDK erweitert werden.
   Wo die fehlende Include-Datei liegt, kriegt man im Zweifelsfall durch Dateisuche
   im SDK-Baum raus. 

7. Wenn man so nach und nach alle Suchpfad-Erweiterunge hinzugefügt hat, kann das
   Projekt fehlerfrei durchkompiliert, aber nicht gelinkt werden.

8. Die oben bei der Projekterstellung eingebundenen Dateien runtime_init_clocks.c rauswerfen,
   dafür das SDK-Vereichnis <pico-sdk>\src\rp2_common\pico_runtime_init einbinden.
   Darin sind 3 C-Dateien, die alle eingebunden werden müssen. Insbesondere "runtime_init.c"
   ist wichtig, da diese Datei das korrekte runtime_init enthält und das weak-Symbol
   aus thumb_crt0.S ersetzt (s. unten).
   Dasselbe dann für runtime.c: runtime.c rauswerfen und das Vz. 
   <pico-sdk>\src\rp2_common\pico_runtime einbinden.

9. Das aktuelle Device-Verktorfile aus den Crossworks-Projektverzeichnis durch ein 
   an das SDK angepasste ersetzt, was im Projektverzeichnis liegt:
   RP2040_VectorsSDK.s
   Das Pico-SDK erwartet teilweise andere Segmentnamen sowie eine Implementation in der
   Art, dass alle ungenutzten IRQs auf einen Dummy-Handler namens "__unhandled_user_irq"
   verweisen müssen.

10.Das aktuelle thumb_crt0.S aus dem Rowley-Source-Verzeichnis durch ein eigenes
   ersetzen, was im Projektverzeichnis liegt. 
   Dieses angepasste thumb_crt0.S enthält Erweiterungen, die das Mutex-Array zur
   Initialisierung aus dem Flash ins RAM kopieren ( analog zum data-Segment )
   und die zur Initialisierung das SDK über einen Aufruf von "runtime_init" ausführt
   ( alles vor dem Aufruf von "main")

11. Zu jedem oben über den Include-Pfad hinzugefügtes headerfile gibt es  mindestens
   ein zugehöriges C-File im selben SDK-Teilbaum, die muss man jezt, am besten durch
   drag&drop als dynamischen folder im Projekt hinzufügen. Die dann ebenfalls hinzugeüfgten
   bazel und CMakeFile dateien kann man gleich wieder rausschmeißen, das ist Ballast.
   Auch das ist aufändig.

12.Die Pin-Definitionsdatei ist evtl namensgleich mit SDK-Inlcude-Dateien, daher empfiehlt es
   sich, die Pin-Definitionsdatei aus 
      $(TargetsDir)/RP2040/pico-sdk/src/boards/include
   ins Projektverzeichnis zu kopieren, ggf umzubenennen und von dort zu includen.

13. To be able to debug the ROM-Code, the ROM has to be included in the memory-mapfile:

      <MemorySegment access="ReadOnly" name="FLASH" size="0x01000000" start="0x10000000" />

    You may use the file "RP2040_M0_0_MemoryMap.xml" from the ProjectDir for that.

14. Wenn man jetzt weitere SDK-Teile einbinden will, müssen diese jeweils indviduell über
    #defines in den Projekteinstellungen konfiguriert werden, z.B.:
    für UART
      PICO_DEFAULT_UART=0
      PICO_DEFAULT_UART_RX_PIN=1
      PICO_DEFAULT_UART_TX_PIN=0
    oder für printf/stdio
      LIB_PICO_STDIO_UART
      LIB_PICO_PRINTF_PICO
      LIB_PICO_STDIO


-------------------------------------------------------------------------------
* rumgemache mit Crossworks V.4.X
-------------------------------------------------------------------------------

A) Passender Target-Adapter für RPI DebugProbe ist "CMSIS-DAP"
   a) in den Einstellungen des Adapters die Speed des Adapters auf irgendwas < 10MHz runterdrehen
   b) in dem Skriptfle PR2040_Target.js in der Zeile
            if (TargetInterface.getProjectProperty("Target") == "RP2040_M0_0" && (TargetInterface.peekWord(0x40004014) & 0x88) == 0x08)
      "Target" durch "Target Processor" ersetzen
   c) Jetzt sollte ein connect möglich sein
